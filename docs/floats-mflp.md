# MFLP float runtime plan

Real floating point for compiled programs, using the interpreter's own
format so compiled BASIC65 behaves like interpreted BASIC65. Decision made
2026-07-04: 5-byte CBM MFLP, no fixed-point variant for now.

## Format

Memory format (5 bytes), identical to C64/C65 BASIC variables:

- byte 0: exponent, excess-128; $00 means the value is exactly 0
- byte 1: mantissa MSB; bit 7 is the SIGN (the leading mantissa 1 is
  implied); bytes 2-4: remaining mantissa, big-endian
- ~9 significant decimal digits, range roughly 1e-38 .. 1.7e+38

Working registers in runtime storage (not zero page):

- `fac` (6 bytes unpacked: exp, 4 mantissa bytes with explicit leading 1,
  sign) + `facround` guard byte for rounding
- `farg` same layout, second operand

## Storage model

- Tagged numeric scalars already exist (tag at slot+0): keep tag 0 =
  16-bit int at +1/+2. Redefine tag 1 = **inline 5-byte MFLP** at +1..+5
  (fits: scalar value area is 8 bytes). The current "tag 1 = pointer to
  literal text" scheme is replaced.
- Float arrays: element size becomes 5 bytes for VAR_TYPE_FLOAT arrays
  (type is known at DIM; no per-element tag). Bounds-check shift changes
  from `asl` (x2) to x5 (asl x2 + add).

## Float literals

The compiler should not carry its own decimal-to-MFLP converter in v1.
Instead: float literals go into a dedicated literal pool as text (deduped
like strings), and the generated program header gains a `fltpool` vector.
`rtinit` walks the pool once and converts each literal in place to 5
bytes using the runtime's own `valflt`. Expression codegen then loads
5-byte constants from the converted pool. One conversion per literal per
run, no compile-time float code, and VAL/READ/INPUT reuse the same parser.

## Runtime routines (all in src\runtime\runtime.asm)

Core (order of implementation):

1. `fldfac`/`fldarg` (unpack from [varptr]/pool), `fstfac` (pack to
   [varptr]), `fmovaf` etc.
2. `float16` int16 -> FAC; `qint` FAC -> int16 (truncate toward zero for
   INT() semantics match; range-check to +/-32767 with OVERFLOW error)
3. `fadd`/`fsub`: align exponents, add/sub mantissas, normalize, round
4. `fmul`: mantissa multiply via the MEGA65 math unit (MULTINA/MULTINB at
   $d770/$d774, 64-bit product at $d778) -- this is the big win over the
   C64 shift-add loop; exponent add, normalize
5. `fdiv`: the math unit also divides ($d76c..) -- use it; else classic
   restoring divide
6. `fcmp`: compare packed values (sign, exponent, mantissa) -> A=$ff/0/1
7. `printflt`: classic FOUT algorithm (digit extraction via repeated
   x10 / /10), matching interpreter spacing/rounding, including
   scientific notation for |x| >= 1e9 or < .01
8. `valflt`: text -> FAC (digits, '.', 'e' exponent, signs); INPUT and
   READ float fields route here

Budget note: runtime is currently $2012-$3188; ~3.5KB of headroom to
progbase. Classic MFLP packages fit in ~2-2.5KB; with the hardware
multiplier replacing the mul loop it is comfortably within budget. If it
ever tightens, progbase moves (one constant in runtime.asm plus the
compiler's header emitters, both backends).

## Compiler changes

- **Type propagation** in expressions: the expression compiler gains a
  current-type state (int/float). Factors set it (int literal/var ->
  int; float literal/var, or `/` division result if we adopt BASIC
  semantics -> float). Binary ops on mixed types emit int->float
  promotion of the int side (`float16`), then the float op. This is the
  largest frontend change; IF conditions and assignment conversions hang
  off it (`float var = int expr` -> float16; `int var = float expr` ->
  qint).
- Float expression temporaries: `fpush`/`fpop` (software stack, 8 x 5
  bytes in runtime storage) replace the pha/pha pattern when a
  subexpression is float-typed.
- PRINT float -> `jsr printflt`; comparisons -> `jsr fcmp` + the existing
  1/0 truth wrappers; ABS/SGN/INT float paths.
- New templates only (text side), regenerated bytes for free; every step
  byte-diffed by the harness.

## Verification

- Extend `basic\floats.bas` with arithmetic, mixed int/float, compare,
  INT/ABS/SGN, VAL/STR$, PRINT formatting edge cases (0, negatives,
  .0001, 1e9 boundary, rounding like 0.1+0.2).
- **Interpreter differential test**: xemu's `-importbas file.txt` types
  and RUNs a BASIC program in the interpreter. Run the same fixture
  interpreted and compiled, dump both screens, diff. That makes "matches
  interpreted BASIC65" a mechanical check, not a judgment call -- worth
  adding to emu-test.ps1 as a `-Interp` mode for float fixtures.

## Order of work

1. Runtime: fldfac/fstfac + float16/qint + fadd/fsub + fcmp; printflt
   for integers-valued floats only; fixture proves store/load/add/print.
2. fmul/fdiv via the math unit; full printflt; valflt + literal pool
   init; retire the tag-1-pointer scheme (storefloatref/printfloatref).
3. Compiler type propagation + promotion/conversion emission + fpush/fpop.
4. Float arrays (element size 5) + READ/INPUT float paths.
5. Interpreter-differential harness mode; broaden fixtures.
